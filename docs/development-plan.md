# 開發進度規劃（Airplane Notify MVP）

根據《README.md》的 W1/W2/W3 分階段目標，以下整理首輪迭代的開發甘特與交付重點。

## 🎯 整體節奏

| 週次 | 目標 | 主要交付 | 風險應對 |
|------|------|----------|----------|
| W1 | 幾何核心與前端骨架 | `computePassEvent` 核心函式、OpenSky 資料正規化、Demo 面板（含倒數 / 停留秒數 / 噪音等級） | 先以靜態樣本驗證 CPA；若 API 範例不足，人工建立 sample states 驗證流程 |
| W2 | 告警體驗與資料串流 | 匿名模式輪詢（10–12s）、演出模式提醒（T-30 / T-10）、錯誤與延遲狀態顯示 | 使用指數回退與緩衝，避免輪詢失敗造成體驗中斷 |
| W3 | 現地驗證與報告 | 三場實測資料、命中/誤報統計儀表板、調參建議 | 於雨天與夜間補捉不同場景，並同步日誌便於回溯 |

## ✅ W1 工作拆解

1. 建立 `web/` 前端專案（Vite + React + TypeScript）。
2. 實作局部 ENU 轉換與 `computePassEvent`，附上 Vitest 單元測試。
3. 建立 Demo 面板：顯示半徑內倒數、停留秒數與噪音等級，對齊 README 的最小展示需求。
4. 產出假資料範例，確保幾何演算法能被前端組件消費。

## 🚧 預備議題

- **OpenSky API 範圍限制**：若匿名速率不足，W2 開始評估 Worker Proxy。
- **噪音等級調整**：W1 先依 README 公式實作，W2 再視實測調整閾值。
- **地圖整合**：Leaflet 與 Web Worker 計畫於 W2 開始接手，避免 W1 過早優化。
